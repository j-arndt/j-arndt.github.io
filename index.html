Step 0 — context

The application & the incident

PolicyDocs is an internal business application that ~1,200 agents and service reps use daily to generate and retrieve policy documents. It is a typical three-tier system: a browser front end, a REST API on an application server, and a SQL Server database — with a nightly batch feed and a synchronous call to an external rating vendor's API.

The job of application support isn't to guess — it's to narrow a vague symptom into one provable cause, then fix it in a way that holds up to an audit. Here's that, step by step.

Step 1 — triage

Stabilize and scope it in the first 30 minutes

  1. Confirmed scope & impact. Which users, which actions, what error rate — reproduced the failure with a test account to confirm it wasn't user-specific.
  2. Checked the health signals. API p95 latency had jumped from ~380 ms to 6.5 s; database connection-pool utilization was pinned at 100%; server CPU and memory were normal — pointing at the data tier, not capacity.
  3. Communicated immediately. Posted a status note and ETA to the business and IT channels, opened INC-4471, and set a 30-minute update cadence. In support, silence is its own incident.
Step 2 — diagnose

Let the logs and the data name the cause

The application log showed the front-line symptom — requests waiting on the connection pool:

# app-server log — 09:22
ERROR PolicyDocs.Api.DocumentController
  System.TimeoutException: The connection pool reached its maximum
  size of 100 and timed out waiting for an available connection.
  at GetPolicyDocument(policyId) ... waited 30000ms

A pool runs out when connections are borrowed faster than they're returned — so the real question is "what is holding connections open?" I queried the database for what was actually running:

-- what is running right now, and for how long?
SELECT r.session_id, r.status, r.wait_type,
       r.total_elapsed_time/1000.0 AS elapsed_s,
       SUBSTRING(t.text,1,90) AS sql_text
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.session_id > 50
ORDER BY elapsed_s DESC;

Two findings stacked on top of each other:

Finding A — a slow query holding connections. The document-fetch query was averaging 4.2 s doing a clustered-index scan. A status filter added in the prior night's release had no supporting index.

Finding B — a connection held across a slow external call. The same code path now made a synchronous call to the vendor rating API (avg 5.8 s) while still holding the database connection. Under morning peak load, connections drained faster than they returned → pool exhausted → timeouts.

Step 2b — root cause

Five whys to the real cause

  1. Submissions failed because API requests timed out.
  2. They timed out because no database connection was available (pool exhausted).
  3. The pool exhausted because connections were held open too long during peak.
  4. They were held open by a 4 s unindexed query and a 5.8 s vendor API call made inside the connection's scope.
  5. That happened now because last night's release added the unindexed status filter and moved the vendor call inside the database transaction.
Root cause: A recent change introduced an unindexed query and held a database connection across a slow external API call. Individually survivable; together, under peak load, they exhausted the connection pool and caused the timeouts. Not a capacity problem — a change problem.
Step 3 — resolve under change control

Fix it without breaking anything else

Because the cause was a change, the fix goes through change control — even in an emergency. That's the discipline that keeps a fix from becoming the next incident.

FieldEmergency change — ECR-2218
Change(1) Add nonclustered index on PolicyDoc(status, created_date); (2) move the vendor API call outside the DB transaction scope; (3) temporarily raise pool max 100→200 as a buffer.
Risk / impactLow. Index is additive (no schema break); code change is isolated to one method; config is reversible. No data migration.
Backout planDrop the index, revert the method to prior build, restore pool config — each independently and within minutes.
ValidationReproduce in staging; confirm p95 latency and pool utilization in production for 60 min post-deploy.
ApprovalsApplication Owner + Change Manager (emergency CAB).

Result after deploy: API p95 latency returned to ~410 ms, submission error rate to 0%, and peak connection-pool utilization settled near 46%.

Step 4a — prevent (the runbook)

Turn the fix into a 6-minute runbook

The first time this happens it's a 72-minute investigation. The next time, anyone on the team resolves it from this runbook in minutes. Writing it down is the job.

Runbook · "App slow / timeouts — DB connection-pool exhaustion"
  1. Detect. Alert fires: connection-pool utilization > 80% for 5 min, or API p95 > 2 s.
  2. Confirm. Run the dm_exec_requests query (above) to list long-running sessions and wait types. Check the vendor API status page.
  3. Mitigate now. If a single query dominates, capture its plan; if the vendor API is slow, enable the circuit-breaker / cached-rate fallback to release connections.
  4. Resolve. Add the missing index (validate the plan flips scan → seek) and/or move external calls outside the connection scope. Raise this as a change (ECR).
  5. Verify. Watch p95 latency, error rate, and pool utilization for 60 min. Confirm with a test transaction.
  6. Close & communicate. Update the incident, post the all-clear, and log the root cause for the weekly report.
Escalation path: App Support (owner, 0–30 min) → Database Administration (if blocking/query-plan, 30–60 min) → Vendor Management (if the external API is the cause) → Application Owner + Change Manager (for the ECR).

Make the application observable

Weekly application-health report — driven by one query
-- daily error rate & p95 latency by application, last 7 days
SELECT app_name,
       CAST(request_time AS date)            AS day,
       COUNT(*)                                AS requests,
       SUM(CASE WHEN status >= 500 THEN 1 ELSE 0 END) AS errors,
       ROUND(100.0*SUM(CASE WHEN status>=500 THEN 1 ELSE 0 END)/COUNT(*),2) AS err_pct,
       PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY latency_ms)
         OVER (PARTITION BY app_name) AS p95_ms
FROM app_request_log
WHERE request_time >= DATEADD(day,-7,SYSUTCDATETIME())
GROUP BY app_name, CAST(request_time AS date);

That single report becomes the Monday stakeholder update: error rate, latency, the five slowest queries, and any change-linked incidents — the "analysis and reporting that informs decisions" this role calls for.
